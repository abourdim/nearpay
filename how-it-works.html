Le principe

Le badge ne sait rien.
Et c'est voulu.

La plupart des gens pensent que le badge contient l'addition. C'est faux. Il contient juste une adresse — comme une plaque de rue qui dit "je suis TABLE-12". Toute la logique est côté serveur.

🏷️ Ce que le badge contient

Un badge NFC NTAG213 de 144 octets. La quasi-totalité est vide. On y écrit une seule chose : une URL sécurisée.

Contenu réel du badge :
https://pay.nearpay.io/t/TABLE-12?sig=a3f9b2c1
C'est tout. 144 octets. Pas d'addition, pas de prix, pas de menu.
Pourquoi c'est mieux :
✅ L'addition change en temps réel (nouvelles commandes)
✅ Le badge ne se "périme" jamais
✅ Aucune donnée sensible sur l'objet physique
✅ Si le badge est perdu, zéro risque

🧠 Ce que le POS contient

Le système de caisse (POS) est le vrai cerveau. C'est lui qui sait ce qui a été commandé, par qui, et à quel prix.

Structure d'une note dans le POS :
NOTE #847 — TABLE-12 ├─ Houmous maison ........... 9,00€ ├─ Tajine agneau ........... 24,00€ ├─ Couscous royal .......... 28,00€ ├─ Thé menthe x4 ........... 16,00€ └─ TOTAL ................... 77,00€ Status: OPEN Ouvert: 20:32 | Serveur: Farid
Quand le client tape son badge → NearPay interroge le POS → récupère cette note → l'affiche sur le téléphone du client.

Questions fréquentes

Les vraies questions techniques

Le lecteur NFC "connaît" la table ? +

Non. Le badge est "bête" — il contient juste une URL avec l'identifiant de la table. C'est le serveur NearPay qui interroge le POS pour retrouver les commandes associées. Le badge lui-même ne stocke aucune information de commande.

Que se passe-t-il si la connexion coupe ? +

Le badge a toujours fonctionné hors-ligne côté lecture (le téléphone lit l'URL localement). La page de paiement nécessite une connexion. Dans les lieux avec WiFi faible (festivals), les lecteurs POS bufferisent les transactions et synchronisent dès la reconnexion.

Comment sait-on que la commande appartient à cette table ? +

Le serveur crée une "note" ou "ticket" dans le POS en début de service et l'associe à l'identifiant du badge (ex: TABLE-12). Toutes les commandes saisies pour cette table s'ajoutent à cette note. NearPay récupère la note via l'ID de table — pas via le badge directement.

Un client peut-il payer pour une autre table ? +

Non. Le token dans l'URL est signé (HMAC-SHA256) et lié au venue. Sans la clé secrète, il est impossible de forger un token valide pour une autre table. De plus, chaque token a un timestamp — il expire après 24h.

Où est stocké le numéro de carte du client ? +

Nulle part dans nos systèmes. Stripe tokenise la carte côté client (Apple Pay / navigateur) avant qu'elle ne quitte le téléphone. NearPay ne reçoit jamais le numéro de carte brut — uniquement un token opaque de Stripe. Certification PCI DSS Level 1.

Que reçoit la cuisine exactement ? +

Quand un client commande depuis son téléphone, la commande est d'abord ajoutée à la note POS, puis transmise à l'écran KDS en cuisine via WebSocket en temps réel. Le cuisinier voit : table, plat, quantité, heure et note spéciale (ex: "sans gluten"). Même chose que si le serveur l'avait saisie manuellement.

Le serveur est-il notifié immédiatement ? +

Oui. Dès que le paiement est confirmé par Stripe (environ 2 secondes), NearPay envoie une notification push FCM / APNs sur le téléphone ou la montre du serveur. La notification indique : table, montant, heure, et si une évaluation a été laissée.

Comment fonctionne le split bill en cas de litige ? +

Chaque paiement de split est un PaymentIntent Stripe distinct avec son propre reçu. En cas de litige, chaque convive peut contester uniquement sa part. La réconciliation côté POS montre le total des parts qui doit correspondre exactement au total de la note.
